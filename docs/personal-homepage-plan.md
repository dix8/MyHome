# 个人主页系统方案文档

更新时间：2026-03-25  
文档目标：为“个人主页 + 后台管理 + 模板中心”项目提供一套可执行的产品与技术方案，重点保证后台管理页面足够好看、易用、可长期扩展。

## 1. 项目定位

本项目不是单纯的静态个人主页，而是一套面向个人品牌展示的内容系统，包含以下 3 个核心部分：

1. 前台主页：对外展示个人信息、作品、技能、经历、社交链接、联系方式等内容。
2. 后台管理：通过管理台维护站点内容、模板、媒体、发布状态。
3. 模板中心：内置多套预设模板，可预览、切换、配置并发布。

项目的核心价值不是“做一个页面”，而是“做一个能长期运营、随时换风格、可持续维护展示内容的个人站系统”。

## 2. 目标与原则

### 2.1 业务目标

1. 用户可以不改代码，通过后台维护主页的主要内容。
2. 用户可以在后台中切换不同视觉模板，而不需要迁移内容。
3. 模板切换前可以预览，切换后可以随时回滚。
4. 后台管理页面要具备明显的设计感，不能是普通后台系统的模板化观感。
5. 系统整体保持轻量，内容结构尽量贴近当前站点的单页展示形态。

### 2.2 设计原则

1. 内容与表现分离：内容结构独立于模板样式。
2. 模板代码化：模板作为代码模块维护，不把整套模板存成数据库字符串。
3. 配置结构化：后台填写的是结构化数据，而不是任意拼接 HTML。
4. 管理体验优先：后台操作路径短、状态清晰、预览直观。
5. 美观不是装饰：后台的视觉设计必须服务于效率、情绪和品牌感。

## 6. 后台视觉与交互方案

这一部分是项目重点。后台必须好看，但不能为了“炫”而损失管理效率。

### 6.1 视觉方向

后台采用“Tech Editorial Dashboard”风格，延续当前站点的技术感，但做得更克制、更高级，而不是传统企业后台风格。

关键词：

1. 精致
2. 克制
3. 有氛围
4. 高级而不浮夸
5. 明确区分主次层级

应避免的问题：

1. 默认感太强的通用后台样式
2. 大量硬边框和廉价阴影
3. 密密麻麻的表格堆叠
4. 过亮纯白背景导致长时间使用疲劳
5. 主按钮、筛选区、卡片区缺乏节奏

### 6.2 视觉语言

视觉策略：

1. 保留当前站点的深色科技基因，但降低前台那种强发光强对比，避免后台使用疲劳。
2. 主背景建议使用石墨黑或深蓝灰，而不是纯黑。
3. 强调色沿用当前站点的 `青蓝 + 紫色` 渐变体系，作为品牌识别。
4. 卡片和表单使用毛玻璃弱化版、细描边、柔和高亮，而不是廉价霓虹光效。
5. 标题和关键数字可继续使用带技术气质的等宽字体点缀，正文保持高可读性无衬线字体。

后台视觉不应复制前台的“炫光网页”，而应做成“沿用同一品牌语言的高级管理台”。

### 6.3 关键交互原则

1. 重要页面必须支持即时预览。
2. 表单页面优先做分组和折叠，而不是无尽滚动。
3. 高频动作固定在同一位置，例如保存草稿、预览、发布。
4. 页面状态要明确区分：未保存、草稿、已发布、已变更。
5. 切换模板时一定要提供视觉对比和兼容性提示。

### 6.4 后台页面体验目标

后台不应只是“能管理”，而应让用户有持续维护内容的欲望。

体验目标：

1. 打开后台时有品牌感，而不是进入一个陌生的通用管理系统。
2. 编辑主页内容时能看到清晰的结构和实时反馈。
3. 切换模板时能像挑选主题一样自然、直观、愉悦。
4. 发布流程要有仪式感，但步骤不能繁琐。

### 6.5 响应式设计定案

你这个项目的重点是“前台必须在电脑端和手机端都很好看”，所以响应式不是补丁，而是从一开始就作为主设计目标。

正式要求如下：

1. 前台站点按双端高质量适配来做，不允许只在桌面端好看。
2. 后台管理按桌面端优先、平板可用、手机可完成轻量操作来设计。
3. 模板开发必须同时提交桌面版和手机版效果，不接受只做单端视觉稿。
4. 模板中心的预览图至少要包含桌面和手机两种预览状态。

设计断点：

1. 手机：`360-430px`
2. 小平板：`768px`
3. 大平板 / 小笔记本：`1024px`
4. 桌面：`1280px+`
5. 大屏：`1440px+`

前台适配原则：

1. Hero 区在桌面端强调氛围和留白，在手机端强调信息聚焦和首屏完成度。
2. 技能区和项目区在桌面端可采用多列卡片，在手机端必须收敛为单列或双列的舒适布局。
3. 联系方式卡片在手机端必须避免过小点击区域。
4. 模板的动效在手机端要自动降强度，避免掉帧和操作迟滞。

后台适配原则：

1. 桌面端使用完整的三栏结构：模块树、编辑区、预览区。
2. 平板端预览区改为抽屉或切换视图。
3. 手机端不强求完整三栏后台，而是改为单列编辑流，保证可维护基本内容。
4. 复杂拖拽和批量编辑优先在桌面端完成，不把后台手机体验做成伪完整功能。

## 8. 模板中心技术方案

模板系统不能只是“换 CSS 皮肤”，而应是有边界、有契约的渲染系统。

### 8.1 核心思路

所有模板都基于同一份统一数据结构进行渲染。

即：

后台内容 -> 标准化数据层 -> 模板渲染器 -> 前台页面

模板不直接读取数据库表，而是只接收统一的 `SiteRenderData`。

### 8.2 模板目录规范

项目统一使用一个专门存放模板的目录，例如：

```text
src/
  templates/
    neon-tech/
      manifest.ts
      schema.ts
      entry.tsx
      styles/
        index.css
        hero.css
        projects.css
      scripts/
        index.ts
        particles.ts
        typing.ts
      components/
        Hero.tsx
        Skills.tsx
        Projects.tsx
        Contact.tsx
      assets/
        preview.png
        bg-grid.svg
    minimal-card/
      manifest.ts
      schema.ts
      entry.tsx
      styles/
      scripts/
      components/
      assets/
```

规范要求：

1. 项目中有且仅有一个专门的 `templates` 根目录。
2. 每个模板必须是 `templates` 下的独立文件夹。
3. 模板自己的样式、脚本、组件、图片都放在自己的模板目录里。
4. 模板内部可以平铺，也可以拆成 `styles/`、`scripts/`、`components/`、`assets/` 子目录。
5. 文件少时可平铺，文件增多后按子目录拆分，但仍必须留在模板目录内部。

### 8.3 模板组成与边界

每个模板至少包含以下文件：

1. `manifest.ts`：模板名称、封面、说明、版本、支持的模块
2. `schema.ts`：模板可配置项定义
3. `entry.tsx`：模板渲染入口
4. `styles/` 或模板样式文件：模板自己的样式
5. `scripts/` 或模板脚本文件：模板自己的动效和交互
6. `components/`：模板专用组件
7. `assets/`：模板专用图片和装饰资源

边界规则：

1. 模板负责页面结构、视觉样式、动画效果和模板专用资源。
2. 系统层负责内容数据、后台管理、媒体库、发布回滚、模板加载。
3. 模板之间禁止互相引用文件。
4. 模板只能引用自己目录下的文件，或者引用 `shared/` 中的公共能力。
5. 粒子背景、打字机、卡片光晕等效果归属于模板，不做站点级全局特效中心。

### 8.4 模板配置能力

模板可配置项包括：

1. 主色与强调色
2. 字体方案
3. 页面宽度
4. Hero 布局样式
5. 粒子背景开关与密度
6. 卡片样式
7. 背景样式
8. 模块分隔方式
9. 按钮风格
10. 打字机标题样式

这些配置项并不是所有模板都必须具备，而是由各模板在自己的 `schema.ts` 中声明，后台按 schema 动态渲染配置表单。

### 8.5 模板注册与加载原则

1. 切换模板不修改原始内容数据。
2. 模板切换只改变模板标识和模板配置。
3. 模板元信息优先从模板目录中的 `manifest.ts` 读取，而不是手工维护一份散乱配置。
4. 系统接入模板时只认固定入口，例如 `manifest.ts`、`schema.ts`、`entry.tsx`。
5. 如模板不支持某模块，则后台给出提示并允许降级处理。
6. 每次模板切换都生成一条可回滚的版本快照。
7. 当前未启用的模板脚本不参与页面运行，按模板按需加载。

## 10. 技术架构正式定案

### 10.1 最终技术栈定案

这套项目不再停留在“推荐”，正式定案如下：

1. 前后台统一框架：`Next.js App Router + React + TypeScript`
2. 后台 UI 技术：`Tailwind CSS + Radix UI + 自定义 Design Tokens`
3. 前台模板实现：`React + 每模板独立 CSS Modules / scoped CSS + 每模板独立脚本`
4. 动效方案：`Motion` 负责 React 过渡动画，模板私有脚本负责粒子、打字机等特效
5. 数据库：`PostgreSQL`
6. ORM：`Prisma`
7. 登录鉴权：`Auth.js`
8. 文件策略：`本地上传 + 自定义 URL`
9. 图片渲染：`next/image`

这是正式选型，不再作为待定项。

文件策略说明：

1. 系统不接入 `S3`、`OSS`、`COS`、`MinIO` 等对象存储。
2. 后台图片字段统一支持两种来源：本地上传文件、手动输入外部 URL。
3. 媒体库只管理本地上传文件，不记录外部 URL 资源。
4. 前台模板只消费最终的图片 URL，不关心该资源来自本地还是外链。
5. 本地上传文件统一保存到项目可控的上传目录，例如 `public/uploads/`。

### 10.2 为什么最终确定这套栈

之所以定这套，而不是继续比较别的方案，核心原因有 5 个：

1. 你要的是“好看”，这意味着后台和前台都需要高自由度定制，不能被通用 CMS 后台样式绑住。
2. 你要的是“模板中心”，这要求模板能独立管理样式、脚本和布局，Next.js 的组件化和动态加载更适合。
3. 你要的是“电脑端和手机端都很好看”，React + Next.js 配合定制样式体系更容易做精细响应式。
4. 你这个项目最终是产品，不只是页面，所以需要一套既能做设计表现，又能做后台管理的统一工程体系。
5. 你当前的媒体使用规模不大，采用“本地上传 + 手动 URL”比对象存储更轻、更直接，也更符合当前项目需求。

### 10.3 前台与后台的实现分工定案

为了保证“好看”不被技术栈拖累，前台和后台的实现方式正式区分如下：

前台站点：

1. 以模板为中心实现视觉表现。
2. 每个模板独立维护自己的布局、样式、脚本、资源。
3. 模板层优先使用 CSS Modules 或模板私有样式文件，不依赖后台那套通用样式。
4. 特效和氛围感由模板自己决定，不做站点级统一特效引擎。

后台管理：

1. 使用统一设计系统实现，避免后台视觉散乱。
2. 底层交互组件使用 Radix UI，视觉层全部重做，不直接沿用默认风格。
3. Tailwind 主要服务后台和系统层，不主导模板层视觉。
4. 后台以编辑效率和品牌感并重，不追求传统 SaaS 控制台样式。

### 10.4 为什么不优先选通用 CMS 默认后台

原因如下：

1. 默认后台往往功能够用，但审美普通，难形成你自己的产品气质。
2. 模板中心、实时预览、主题切换等交互做深后，定制后台更自然。
3. 你的项目是“个人品牌系统”，管理台本身也是产品的一部分。

### 10.5 前台视觉实现定案

为了保证你强调的“好看”，前台视觉不采用通用模板站常见的平庸做法，而采用以下正式方案：

1. 每个模板单独定义自己的视觉语言，不共用一套模板外观。
2. 模板内部使用自己的颜色、排版、卡片、背景、动效方案。
3. 桌面端优先营造氛围感和层次感，手机端优先保证节奏和阅读流畅。
4. 动效采取“少而准”的方式，不靠堆很多动画制造高级感。
5. 手机端不简单缩放桌面布局，而是按手机信息密度重新排布。

模板的审美要求：

1. 至少一套延续你当前站点的深色技术风。
2. 至少一套偏极简、高级感的卡片模板。
3. 至少一套强调个人品牌和作品展示的表达型模板。

### 10.6 后台视觉实现定案

后台正式采用“定制化创作者工作台”的实现方式，而不是企业 OA 风或传统 CMS 风。

明确要求：

1. 后台视觉与前台品牌气质保持关联，但不照搬前台夸张特效。
2. 后台采用更克制的深色或中性色体系，减少长时间使用疲劳。
3. 所有关键页面都要有明确的信息层级、留白和节奏。
4. 表单、模板中心、预览区必须是重点设计页面。
5. 后台的“好看”优先来自版式、比例、字体、层次，不来自廉价发光和阴影。

## 16. 最终结论

这类项目最容易做偏的地方，是一开始把重点放在“前台长什么样”，而忽略了后台的产品感。  
真正长期可用的方案应该是：

1. 前台做成模板化展示系统
2. 后台做成定制化创作者工作台
3. 内容数据与模板彻底解耦
4. 模板切换走预览、发布、回滚的完整链路

后台“足够好看”的实现方式，不是选一个现成后台框架皮肤，而是从一开始就把后台当成一个需要设计语言、节奏、品牌感和编辑体验的产品来做。
